# Pack template — the canonical `context-pack.md` layout

`prepare-for-dev` writes one `spec`-role artifact named `context-pack.md` onto the
group's **container** node (the release/phase). Use this exact section order. Section 0
(the verify-first preamble) is **copied verbatim** into every pack — it is the
instruction plan mode follows. Everything else is synthesised from the members.

Conventions:
- Tag every code-level claim you can't confirm from the graph alone with **`[VERIFY]`**
  and tie it to a concrete code location/assumption. You are asserting, not confirming —
  the live-code check is plan mode's job.
- The **canonical** acceptance for each leaf is its node `done_looks_like`. The acceptance
  index below is a **dated snapshot** for reading convenience; if they disagree, the node
  wins.
- Refer to members by **Haven ref** (e.g. `HV-3`), never by a pack-local id — there is no
  second backlog.

---

## Skeleton (fill the placeholders; keep section 0 verbatim)

```markdown
# Context pack — <group title>  (<CONTAINER-REF>)
_Generated by prepare-for-dev on <DATE>. Members: <REF, REF, …>._

## 0. VERIFY FIRST — read this before planning or writing any code

This pack describes work that does **not exist yet**. Treat every statement below as an
**assumption**, not a fact. Before you plan or build:

1. For each claim tagged **[VERIFY]**, check it against the live code now.
2. **If an assumption is wrong but the brief is sound** (a `done_looks_like` is off, a
   dependency is missing or spurious, a shared contract is slightly different): make
   correcting Haven the **first step of your plan**, and get it approved before building —
   e.g. `haven item update <ref> --done-looks-like "…"` or `haven depend <a> --on <b>`.
   Fix the **graph**, not just this doc or the plan file. Never silently build to a wrong
   target.
3. **If the brief is structurally wrong** (a leaf is mis-scoped or missing, the
   decomposition is off, the group is incoherent): **STOP. Do not patch this pack.**
   Re-invoke `orchestrate-plan` on the affected branch, let it restructure (archive +
   lineage), then regenerate this pack.
4. This same pack is your **doneness yardstick**: a leaf is done when its `done_looks_like`
   — plus any shared requirement it inherits from section 2 — is demonstrably met. The
   canonical acceptance is the node's field, not the snapshot in section 4.

## 1. Foundation — why this group exists
<the shared purpose / why, in a few lines. What is this batch delivering and for whom.>

## 2. Cross-cutting requirements & shared behaviour  (write once; every member inherits)
<the material true for the WHOLE group, not one leaf: shared interface/API contracts,
data model, naming/error conventions, security/permission rules, behavioural scenarios
several members must satisfy. This is the write-once content that justifies a pack.
Tag code-level claims [VERIFY].>

## 3. External dependencies — the group's boundary
<for each dependency that lands OUTSIDE this group:
- DONE deps: name them + the output/acceptance the group can rely on (read-only context).
- UNBUILT deps: name them + status + "pull into the group, or it blocks <ref>". A decision
  for the human — this pack does not auto-expand scope.>

## 4. Per-leaf acceptance index  (snapshot of <DATE> — node `done_looks_like` is canonical)
- **<REF>** — <title>: <done_looks_like>. Depends on: <refs or none>. [VERIFY: <assumption>]
- **<REF>** — …
```

---

## When NOT to write a pack

If the shared-context assessment (SKILL flow step 5) finds the members share no
architecture, contracts, data model, or sequencing, do **not** synthesise a pack. Record
the call instead, so a later session doesn't re-litigate it:

`haven artifact add <CONTAINER-REF> --role decision --name batch-decision.md --content
"Simple batch — no shared architecture found; dispatch members individually."`
